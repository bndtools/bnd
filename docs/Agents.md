# Agents Guide for docs

This file defines agent workflows for the documentation site under `docs/`.

## Purpose

- Maintain the bnd manual site generated by Jekyll with the Just-the-Docs theme.
- Keep docs navigation, search, release-versioned pages, and generated reference content coherent.

## Prerequisites

- Preferred: Docker Desktop/Engine with `docker` CLI available
- Check Docker availability before docs work:
  - `docker --version`
  - `docker compose version` (if compose workflows are used)
- Fallback only when Docker is unavailable: Ruby + Bundler environment compatible with repository Gemfile

## JTD Template and Layout Usage

- Theme and core behavior configured in `_config.yml` (`theme: just-the-docs`).
- Reusable UI/layout pieces live in:
  - `_layouts/` (for page layouts such as `bnd.html`)
  - `_includes/` (header, nav, sidebar, footer, head customizations)
- Prefer layout/include reuse over page-local HTML duplication.

## NavBar, Search, Static Links, Glossary

- Navigation is driven by frontmatter and Just-the-Docs collection settings in `_config.yml`.
- Top Nav / static links are configured with `aux_links` in `_config.yml`.
- Search uses Just-the-Docs/Lunr integration (`search_enabled: true`).
- Glossary-like discoverability should be implemented with stable collection pages and explicit cross-links, not ad-hoc isolated pages.

## Versioned Releases Support

- Old releases are hosted under `docs/releases/<version>/`.
- For release doc generation, set release metadata in `_config.yml`:
  - `releasename: <version>`
  - `baseurl: /releases/<version>` (when generating release docs)
- Keep `master` branch config pointed at current docs state after release branch operations.
- Follow process guidance in `ADDING_RELEASE_DOCS.md` and related release documentation notes.

## Generated Reference Content

- `_heads` and `_instructions` are generated from bnd syntax sources via the CLI `generatemanual` command.
- Use `generate.sh` in `docs/` after building bnd CLI jar.
- Use `_ext` folders to append/override generated markdown frontmatter/content when needed.

## Build and Validation

- Preferred local development server (Docker):
  - `docker run --rm -it -p 4000:4000 -v "$PWD":/srv/jekyll -w /srv/jekyll jekyll/jekyll:4 jekyll serve -w --incremental --host 0.0.0.0`
- Preferred build validation (Docker):
  - `docker run --rm -it -v "$PWD":/srv/jekyll -w /srv/jekyll jekyll/jekyll:4 jekyll build`
- Fallback local scripts (only when Docker is not available):
  - `./run.sh`
  - `./build.sh`
- Validate in browser:
  - nav integrity
  - search results quality
  - internal/static links
  - rendering of generated collections and release pages

## GitHub Pages Hosting Expectations

- Site is authored in markdown, built by Jekyll, and hosted with GitHub Pages conventions.
- Keep config/plugins compatible with GitHub Pages-compatible Jekyll workflows used by this repo.
- When changing docs build behavior, ensure CI and local scripts still produce equivalent site output.
